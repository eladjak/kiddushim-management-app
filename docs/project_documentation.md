# אפליקציית קידושישי - דוקומנטציה מפורטת

## מבוא ומטרת המערכת

מערכת קידושישי הינה פלטפורמה מקיפה לניהול ארגון התנדבותי המאפשרת ניהול משתמשים, אירועים, ציוד, דיווחים ומשובים בעברית עם תמיכה מלאה ב-RTL. המערכת פותחה במקור באמצעות Lovable אך נדרשת הסבה לפיתוח מתקדם יותר עם שליטה מלאה בקוד.

## ארכיטקטורת המערכת

### טכנולוגיות מרכזיות
- **צד לקוח**: React + TypeScript, Vite, Tailwind CSS, shadcn-ui
- **ניהול מצב**: React Query
- **ניהול נתונים**: Supabase (אחסון, אימות, אחסון קבצים)
- **ניתוב**: React Router
- **טפסים**: React Hook Form + Zod
- **עיצוב**: Tailwind + shadcn-ui components

### מבנה הקוד
המערכת מאורגנת במבנה תיקיות מודולרי:

```
src/
├── components/    # קומפוננטות React מופרדות לפי הקשר
├── context/       # קונטקסט React (בעיקר אימות)
├── data/          # נתונים סטטיים 
├── hooks/         # הוקים מותאמים אישית
├── integrations/  # אינטגרציות עם שירותים חיצוניים (Supabase)
├── lib/           # פונקציות שירות
├── pages/         # דפי האפליקציה
├── styles/        # סגנונות גלובליים
├── types/         # טיפוסי TypeScript
└── utils/         # פונקציות עזר
```

## מודלים ומבנה נתונים

המודלים העיקריים במערכת:

### משתמשים ופרופילים
- `profiles`: משתמשי המערכת עם מידע כמו שם, תפקיד, אימייל, טלפון, תמונה
- תפקידים: מנהלים, רכזים, מתנדבי נוער, בנות שירות, ספקי תוכן

### אירועים
- `events`: אירועים עם מידע על מיקום, תאריך, זמני הקמה/פעילות/פירוק
- `event_assignments`: שיוך משתמשים לאירועים עם תפקידים
- `event_equipment`: ציוד המשויך לאירועים

### ציוד
- `equipment`: פריטי ציוד הזמינים לשימוש בארגון
- `equipment_changes`: שינויים ובקשות לשינויים בציוד

### דיווחים ומשובים
- `reports`: דיווחים על אירועים
- `feedback`: משובים על פעילויות ואירועים

### תקשורת
- `messages`: הודעות בין משתמשים
- `notifications`: התראות למשתמשים

## תהליכים עיקריים

### ניהול משתמשים ואימות
1. המערכת משתמשת ב-Supabase Auth לאימות משתמשים
2. מנגנון הרשאות מבוסס תפקידים (`user_role`)
3. העלאת תמונות פרופיל ושמירתן ב-Supabase Storage

### ניהול אירועים
1. יצירה ועריכה של אירועים
2. הקצאת מתנדבים ותפקידים לאירועים
3. שיוך ציוד לאירועים
4. דיווחים ומשובים לאחר אירועים

### ניהול ציוד
1. ניהול מלאי ציוד
2. בקשות לשינויים בציוד ואישורן
3. מעקב אחר מצב וזמינות הציוד

## אינטגרציות

### Supabase
- **אחסון נתונים**: כל טבלאות המידע
- **אחסון קבצים**: תמונות פרופיל, תמונות אירועים
- **אימות**: מערכת כניסה והרשמה
- **RLS (Row Level Security)**: אבטחת נתונים ברמת השורה

### מפות
- שימוש ב-mapbox-gl להצגת מיקומי אירועים

## אתגרים וחולשות עיקריות

1. **תמיכה בעברית**: המערכת מתמודדת עם אתגרים בקידוד עברית, בעיקר בתקשורת עם Supabase
2. **תלות ב-Lovable**: מבנה הקוד הנוכחי קשור לפיתוח ב-Lovable ומוגבל ביכולת להתרחב
3. **ניהול מצב**: חסר מנגנון מרכזי וקוהרנטי לניהול מצב המערכת
4. **טיפול בשגיאות**: מוגבל ולא עקבי

## הערות ומסקנות

המערכת במצבה הנוכחי מספקת בסיס איתן ליכולות הניהול הנדרשות, אך מוגבלת על-ידי הפלטפורמה שבה פותחה. המעבר לפיתוח עצמאי יאפשר גמישות רבה יותר, שליטה טובה יותר ויכולת להוסיף תכונות מתקדמות שלא היו אפשריות קודם.

הדוקומנטציה הזו משקפת את המצב הנוכחי של המערכת ותשמש כבסיס לתכנון השדרוגים העתידיים.
