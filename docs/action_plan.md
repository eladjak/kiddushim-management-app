# תכנית פעולה - פרויקט קידושישי

## סיכום המצב הנוכחי

פרויקט קידושישי הוא מערכת לניהול ארגון התנדבותי, שכוללת ניהול משתמשים, אירועים, ציוד, דיווחים ומשובים. המערכת פותחה במקור באמצעות פלטפורמת Lovable, אולם כעת נדרש לשדרג אותה ולעבור לפיתוח עצמאי ללא מגבלות הפלטפורמה.

### אתגרים מרכזיים

1. **תלות בפלטפורמת Lovable** - מגבילה את יכולת ההרחבה והשליטה המלאה בקוד
2. **ניהול מצב לא עקבי** - חוסר במערכת מרכזית לניהול מצב האפליקציה
3. **תמיכה בעברית** - אתגרים בטיפול בטקסט עברי ותמיכה ב-RTL
4. **הרשאות והגנה** - חוסר במערכת הרשאות גרנולרית ומתקדמת
5. **ממשק משתמש** - צורך בשיפור ועדכון הממשק לסטנדרטים מודרניים

## תכנית השדרוג המקיפה

תכנית הפעולה מחולקת ל-5 שלבים עיקריים, עם אומדן זמן כולל של 13-18 שבועות:

### שלב 1: הכנה וארגון מחדש של הקוד (2-3 שבועות)

#### 1.1 סידור מבנה הפרויקט
- **המטרה**: ארגון מחדש של התיקיות על פי עקרונות הפרדת אחריות
- **משימות מרכזיות**:
  - יצירת מבנה תיקיות חדש לפי המפרט בקובץ "directory_structure.md"
  - העברת קבצים למיקומם החדש
  - עדכון ייבואים בכל הקבצים

#### 1.2 ניקוי תלויות ועדכון חבילות
- **המטרה**: הסרת תלויות ב-Lovable ועדכון חבילות
- **משימות מרכזיות**:
  - עדכון package.json וניקוי תלויות מיותרות
  - הוספת חבילות חדשות הנדרשות לשדרוג
  - בדיקת תאימות בין החבילות השונות

#### 1.3 הגדרת תצורה וסביבת פיתוח
- **המטרה**: יצירת סביבת פיתוח מותאמת אישית
- **משימות מרכזיות**:
  - הגדרת קבצי .env וסביבות עבודה
  - התאמת קבצי קונפיגורציה
  - הגדרת סביבות CI/CD בסיסיות

### שלב 2: שדרוג תשתיות ומערכת ניהול מצב (3-4 שבועות)

#### 2.1 שדרוג מערכת האימות וניהול משתמשים
- **המטרה**: יצירת מערכת אימות מתקדמת כמתואר ב-"auth_permissions.md"
- **משימות מרכזיות**:
  - יצירת שכבת אבסטרקציה מעל Supabase Auth
  - פיתוח מערכת הרשאות RBAC
  - הטמעת קומפוננטות RequireAuth ו-RequirePermission
  - הגדרת RLS ב-Supabase לתמיכה בהרשאות

#### 2.2 שדרוג מערכת ניהול מצב
- **המטרה**: יישום ארכיטקטורת ניהול מצב מבוססת React Query כמתואר ב-"state_management.md"
- **משימות מרכזיות**:
  - הגדרת QueryClient מרכזי
  - יצירת שכבת שירותים לכל ישות
  - פיתוח custom hooks עבור queries ו-mutations
  - טיפול במצבי טעינה ושגיאות

#### 2.3 שיפור התמיכה בעברית
- **המטרה**: טיפול בבעיות קידוד ותצוגת טקסט עברי
- **משימות מרכזיות**:
  - יצירת מידלוור לטיפול בטקסט עברי
  - הוספת תמיכה ב-RTL במרחב האפליקציה
  - טיפול בתוויות ממשק בשפה העברית

### שלב 3: שדרוג ממשק משתמש וחוויית משתמש (3-4 שבועות)

#### 3.1 שיפור רכיבי UI
- **המטרה**: עדכון רכיבי ממשק המשתמש כמתואר ב-"ui_ux_improvements.md"
- **משימות מרכזיות**:
  - התאמת רכיבי shadcn-ui לתמיכה מלאה ב-RTL
  - יצירת טפסים מתקדמים עם ולידציה
  - הוספת מצבי טעינה ורכיבי Skeleton loaders

#### 3.2 שדרוג דפים מרכזיים
- **המטרה**: עיצוב מחדש של דפי האפליקציה המרכזיים
- **משימות מרכזיות**:
  - עיצוב מחדש של הדף הראשי והדשבורד
  - שיפור ממשק ניהול אירועים ולוח שנה
  - שיפור ממשק ניהול ציוד ומשתמשים

#### 3.3 שיפור UX
- **המטרה**: הוספת אנימציות ומעברים, שיפור זמני תגובה
- **משימות מרכזיות**:
  - הוספת אנימציות מעבר בין דפים
  - שיפור משוב למשתמש והתראות
  - אופטימיזציה של זמני טעינה

### שלב 4: פיתוח תכונות חדשות ושיפורים פונקציונליים (3-4 שבועות)

#### 4.1 מערכת התראות מתקדמת
- **המטרה**: פיתוח מערכת התראות בזמן אמת
- **משימות מרכזיות**:
  - יצירת קונטקסט התראות מרכזי
  - פיתוח רכיבי התראות חזותיות
  - הגדרת מערכת הפצת התראות

#### 4.2 שיפור מערכת הדיווחים והמשובים
- **המטרה**: שדרוג מערכת הדיווחים וויזואליזציה של נתונים
- **משימות מרכזיות**:
  - יצירת דוחות אנליטיים מתקדמים
  - הוספת גרפים וויזואליזציות נתונים
  - פיתוח יכולת ייצוא דוחות

#### 4.3 אופטימיזציה ושיפורים טכניים
- **המטרה**: שיפור ביצועים ואמינות האפליקציה
- **משימות מרכזיות**:
  - אופטימיזציה של queries ו-mutations
  - שיפור טיפול בשגיאות
  - הוספת לוגים וניטור

### שלב 5: בדיקות, תיעוד והטמעה (2-3 שבועות)

#### 5.1 בדיקות ואבטחת איכות
- **המטרה**: הבטחת איכות ואמינות האפליקציה
- **משימות מרכזיות**:
  - כתיבת בדיקות ליחידות קוד מרכזיות
  - בדיקות אינטגרציה
  - בדיקות משתמש

#### 5.2 תיעוד מקיף
- **המטרה**: יצירת תיעוד מפורט למפתחים ומשתמשים
- **משימות מרכזיות**:
  - תיעוד API פנימי
  - מדריך משתמש מקיף
  - עדכון README ותיעוד התקנה

#### 5.3 הטמעה והדרכה
- **המטרה**: פריסת האפליקציה והדרכת משתמשים
- **משימות מרכזיות**:
  - העברה לסביבת ייצור
  - הדרכת משתמשי קצה
  - איסוף משוב ותיקונים ראשוניים

## עקרונות עבודה מנחים

1. **עבודה איטרטיבית** - פיתוח בשלבים קטנים עם בדיקות שוטפות
2. **גמישות** - התאמה מתמדת של התכנית לפי הצרכים המשתנים
3. **תקשורת** - משוב קבוע ממשתמשים ובעלי עניין
4. **אבטחת איכות** - בדיקות שוטפות לאורך כל תהליך הפיתוח

## צעדים מיידיים

1. **הקמת סביבת פיתוח** - הקמת סביבת פיתוח מעודכנת עם כל החבילות הנדרשות
2. **גיבוי** - יצירת גיבוי מלא של הקוד והנתונים הקיימים
3. **תכנון ארכיטקטורה מפורט** - השלמת תכנון מפורט של ארכיטקטורת האפליקציה החדשה
4. **פתיחת Github Issues** - הגדרת המשימות הראשונות במערכת ניהול המשימות

## סיכום

המעבר מ-Lovable לפיתוח עצמאי מהווה הזדמנות מצוינת לשדרוג מהותי של האפליקציה. תכנית העבודה המפורטת כאן מאפשרת מעבר הדרגתי ובטוח, תוך שיפור משמעותי ביכולות המערכת, חווית המשתמש והאבטחה. הפרויקט מחולק לשלבים ברורים, עם משימות מוגדרות, מה שמאפשר גמישות וניהול סיכונים יעיל. 

בסופו של תהליך זה, אפליקציית קידושישי תהיה משודרגת, מאובטחת יותר, ויעילה יותר, עם ממשק משתמש מודרני ומערכת תשתית איתנה שתאפשר הוספת יכולות חדשות בעתיד.
